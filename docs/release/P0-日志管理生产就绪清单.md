## 开发阶段放宽的条件（上线前需修复）

### 1. 日志轮转与文件管理

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **文件日志轮转** | 文件输出使用 `os.O_APPEND` 无轮转，单文件无限增长 | 集成 `lumberjack` 或类似库，支持按大小/时间轮转、压缩、保留份数 |
| **文件句柄泄漏** | `New()` 的 file 模式打开文件后永不关闭（`zerolog.Logger` 不实现 `io.Closer`） | 实现 `Close()` 方法管理文件生命周期，或在 Logger 结构体中持有文件引用 |
| **文件路径安全** | 默认为空时静默不写入 | 校验路径有效性，不可写时 fallback 到 stderr 并告警 |

**方案**: 已自实现 `dailyWriter`（`rotate.go`） — 按日期切分 + 同日内超大小滚多个文件 + 按天过期清理。不使用 lumberjack，自用服务日志量不大，按天组织更直观。

> ⚠️ 如果需要 gzip 压缩旧文件或 SIGHUP 手动轮转，可考虑迁移到 lumberjack。

### 2. 采样与限流

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **日志采样** | 全量输出，高频错误日志可能打满磁盘 | 实现 error 级别以上全量，info/debug 级别采样（每 N 条记录 1 条） |
| **panic 恢复** | 日志模块本身 panic 无兜底 | 所有公共方法加 `recover()`，确保日志失败不影响业务 |

### 3. 远程日志收集

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **日志采集** | 仅本地文件/stdout | 配置 Loki/Promtail 或直接 HTTP 推送（日志 agent sidecar） |
| **结构化字段规范** | `caller` 使用完整路径 | 生产环境需统一字段命名规范，使用相对路径或 package+函数 |

### 4. Context 链路追踪

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **Gin 中间件注入 TraceID** | 中间件不自动生成 TraceID | 实现 Gin 中间件：请求入口生成 TraceID → 注入 Context → Logger 自动提取 |
| **SpanID 传播** | 仅支持手动设置 | 实现跨服务 SpanID 传播（从请求头提取或生成） |

### 5. 审计日志持久化

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **审计日志存储** | 仅作为普通日志行输出，无独立存储 | 审计日志应写入独立文件或数据库表，确保不可篡改（append-only + 定期归档） |
| **审计日志格式** | 嵌入在业务日志流中 | 配置独立输出目标（文件/syslog/数据库），与业务日志分离 |

### 6. 性能与基准

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **性能基准** | 无基准测试 | 实现 `*_bench_test.go`：测量千条/万条并发日志的吞吐量和 P99 延迟 |
| **异步日志** | 同步写入，I/O 可能阻塞业务 | 高并发场景可配置异步写入（channel + worker pool，批量 flush） |
