# P0 — 统一数据访问层生产就绪清单

> 当前 `internal/database/` 为快速迭代开发所做的妥协，**生产上线前必须修复**。

---

## 已实现的功能

| 功能 | 状态 | 说明 |
|------|------|------|
| MySQL 连接池管理 | ✅ | `database/mysql/` — `sql.Open` + Ping 校验 + 池参数配置 |
| Redis 客户端管理 | ✅ | `database/redis/` — `go-redis/v9` + Ping 校验 + 池大小配置 |
| DBManager 全局管理器 | ✅ | `database/manager.go` — InitAll / CloseAll / Health |
| 健康检查聚合 | ✅ | `Health()` 返回所有实例状态，s0/health 集成 |
| 多实例配置 | ✅ | `config.yaml` 支持 main/nas 等多实例 |
| 环境变量覆盖 | ✅ | `APP_DATABASE_MYSQL_MAIN_*` 系列变量 |

## 开发阶段放宽的条件（上线前需修复）

### 1. 已实现但需要生产加固的存储类型

| 存储 | 驱动 | 当前状态 |
|------|------|----------|
| **PostgreSQL** | `pgx/v5/pgxpool` | ✅ 连接池管理 + Ping 健康检查 + 池参数配置，接入 DBManager |
| **MinIO** | `minio-go/v7` | ✅ S3 客户端 + bucket 校验 + 健康检查，接入 DBManager |
| **RustFS** | `minio-go/v7` (S3 兼容) | ✅ 同 MinIO 实现，独立包名 + 实例管理，接入 DBManager |

**生产加固要求**: 三项均为基础实现，需统一参见下方 TLS/SSL、重连、监控等通用整改项。

### 2. 连接管理欠缺

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **自动重连** | 依赖 `database/sql` 和 `go-redis` 内置的重连机制，应用层无重试 | 实现应用层重试：连接断开时自动重试 N 次（指数退避），超过阈值后标记为 unhealthy |
| **熔断** | 无熔断机制，数据库故障时所有请求直接失败 | service 层集成 circuit breaker（如 hystrix-go 或 gobreaker），数据库故障时快速失败而非 hang |
| **连接池监控** | 未暴露 Prometheus 指标 | 暴露 `db_connections_open` / `db_connection_errors` / `db_query_duration` 等指标（依赖 monitor 模块） |
| **超时控制** | 无统一的 context 超时传递 | 所有数据库查询必须接受 `context.Context` 参数，并设置合理的超时 |

### 3. 安全缺口

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **TLS/SSL** | 数据库连接均为明文（mysql: 不启用 TLS，redis: 无 TLS） | 配置支持 TLS：`config.yaml` 增加 `tls_enabled` / `tls_ca` / `tls_cert` / `tls_key` 字段 |
| **密码来源** | 仅支持 `.env` 环境变量注入 | 生产必须使用 Docker Secrets（`/run/secrets/`）或 Vault，参考 `docs/release/P0-密钥与Secrets管理.md` |
| **密码轮换** | 不支持连接凭据热更新 | 实现密码变更后优雅重连（先建新连接，再切换，再关旧连接） |

### 4. 配置热加载不支持

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **数据库配置热加载** | `config.Config` 支持 `Reloader` 接口，但 `database.InitAll` 未注册 | 实现 `database.Reload()` 方法：检测配置变更→创建新连接池→原子替换→关闭旧连接池（零停机） |
| **配置校验** | 无 `Validate()` 方法 | 实现 `database.Validate(cfg)`：检查必填字段、端口范围、连接可达性 |

### 5. 测试覆盖不足

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **MySQL 集成测试** | 仅有单元测试（mock），无真实数据库连接测试 | 实现 `mysql_integration_test.go`：使用 dockertest 启动真实 MySQL，测试连接/断线重连/池耗尽 |
| **Redis 集成测试** | 仅有单元测试（miniredis） | 实现 `redis_integration_test.go`：使用 dockertest 启动真实 Redis |
| **并发安全测试** | 无并发测试 | 实现 `manager_concurrent_test.go`：100 goroutine 并发 Health() / CloseAll() / Get()，`-race` 通过 |

### 6. 其他

| 问题 | 现状 | 上线要求 |
|------|------|----------|
| **连接池默认值** | 硬编码 `MaxOpen=25`, `MaxIdle=5` | 默认值应在代码中常量定义，并添加注释说明选择依据。生产环境应根据压测结果调整 |
| **DSN 构建** | 简单字符串拼接，不含 query 参数校验 | 使用参数化 DSN 构建器，校验特殊字符（如密码含 `@` 需 URL 编码） |
| **迁移未集成** | `cmd/migrate/` 和 `migrations/` 均为空 | 参考 `docs/module-design.md` §10 实现 golang-migrate 集成，参见 Weekly 2-4 第 12 项 |

## 修复优先级

| 修复项 | 优先级 | 建议阶段 |
|--------|--------|----------|
| TLS/SSL + 安全密码源 | **P0 阻塞** | Phase 1 结束前 |
| 连接池监控指标 | P1 | Phase 3（与 monitor 模块同步） |
| 自动重连 + 熔断 | P1 | Phase 3 |
| 配置热加载 | P1 | Phase 3 |
| PostgreSQL / MinIO / RustFS 生产加固 | P2 | Phase 3-5（取决于 NAS 模块上线时间） |
| 集成测试补充（dockertest） | P2 | Phase 2 |
| 迁移集成 | P2 | Phase 2（与 auth/user 模块同步） |
