# P1 — 发布流水线

CI/CD 仅在手动打 tag 时触发，日常 push 不执行流水线。

---

## 触发方式

```bash
# 确保所有变更已提交
git add -A && git commit -m "feat: xxx"

# 运行发布脚本
./scripts/ci.sh v0.2.0
```

## 流水线流程

```
打 tag v0.x.x → push tag → GitHub Actions
                            ├── lint (golangci-lint)
                            ├── test (go test -race -cover)
                            └── build (docker build + push ghcr.io)
```

## 发布前检查

| # | 检查项 | 开发期现状 | 发布要求 | 涉及文件 |
|---|--------|-----------|---------|----------|
| 1 | 流水线触发条件 | 仅 tag `v*` 触发 | 日常开发 push 不会触发，避免浪费 CI 配额 | `.github/workflows/ci.yml` |
| 2 | 镜像仓库 | 开发期不推送 | 需确保 GitHub Packages (ghcr.io) 已启用，或更换为 Docker Hub | `.github/workflows/ci.yml` |
| 3 | 仓库权限 | 无需配置 | GitHub 仓库需开启 Actions 和 Packages 写入权限 | Settings → Actions → General |
| 4 | Secrets 管理 | 无需 | 生产构建需添加 DOCKER_PASSWORD / GH_PAT 等 secrets | GitHub Secrets |
| 5 | 测试覆盖率 | 仅生成 coverage.out | 发布流水线应上传覆盖率报告，设置质量门禁 | `.github/workflows/ci.yml` |
| 6 | 镜像标签策略 | 推 `v*` 和 `latest` | 生产环境建议固定版本部署，避免 `latest` 意外覆盖 | `.github/workflows/ci.yml` |
| 7 | 版本号规则 | `v0.1.0` | 遵循语义化版本 `vMAJOR.MINOR.PATCH`，与 `docs/development-plan.md` 的版本规划一致 | `scripts/ci.sh` |
| 8 | 版本注入 | 无 | release 构建应通过 `-ldflags` 注入 `main.Version`，使 `/health` 可查询运行版本 | `scripts/build.sh` |

## 本地调试流水线

```bash
# 模拟 CI 流程本地执行
make lint && make test && make release
```

## 快速验证

```bash
# 检查 CI 文件语法（需安装 action-validator 或直接 push 测试）
# 本地模拟 tag 推送（不会真的 push）
git tag v0.1.0-test && git tag -d v0.1.0-test

# 确认脚本可执行
bash scripts/ci.sh v0.1.0  # 会提示工作区不干净（正常）
```
