## 二、基础设施层缺失分析

### 2.1 分布式追踪 (P1)

| 项目 | 说明 |
|------|------|
| **现状** | 仅 `middleware/logger.go` 注入 trace-id (UUIDv4)，无 OpenTelemetry 集成 |
| **影响** | 单体阶段可接受，微服务拆分后无法跨服务追踪请求链路 |
| **建议** | 引入 `go.opentelemetry.io/otel`，当前可先定义 `Tracer` 接口，待微服务化时再接入具体后端 (Jaeger/Tempo) |
| **优先级** | P3 — 单体阶段暂不阻塞，但接口需预留 |

### 2.2 API 文档 (P2)

| 项目 | 说明 |
|------|------|
| **现状** | 无 OpenAPI/Swagger 文档，模块文档手动维护在 `docs/` |
| **影响** | 前端/客户端开发需阅读源码或手动文档 |
| **建议** | 集成 `github.com/swaggo/swag` + `gin-swagger`，在 handler 上添加 `// @Summary` 注解自动生成 OpenAPI 规范 |
| **优先级** | P2 — 适合在模块稳定后统一添加 |

### 2.3 特性开关 / Feature Flags (P3)

| 项目 | 说明 |
|------|------|
| **现状** | 无特性开关系统，功能发布依赖代码部署 |
| **影响** | 无法灰度发布、A/B 测试、渐进式上线 |
| **建议** | `development-plan.md` 中已规划 Phase 6，可参考 `github.com/thomaspoignant/go-feature-flag` 或自建简单 toggle 配置 |
| **优先级** | P4 — 接入 Dify/N8n 时才需要 |

### 2.4 服务发现 / 注册中心 (P4)

| 项目 | 说明 |
|------|------|
| **现状** | 硬编码配置，无服务发现 |
| **影响** | 当前单体架构无影响，微服务化后需要 |
| **建议** | 保持现状，微服务拆分时引入 Consul/etcd |
| **优先级** | P4 — 非当前阶段需求 |

### 2.5 WebSocket / SSE 实时推送 (P2)

| 项目 | 说明 |
|------|------|
| **现状** | 无实时通信能力，所有请求均为 HTTP 请求-响应模式 |
| **影响** | 无法实现实时通知、任务进度推送、AI 流式输出 (SSE) |
| **建议** | 添加 `internal/services/ws/` 包，提供 WebSocket hub + SSE 支持。AI 模块的 `/chat` 流式 SSE 输出依赖此能力 |
| **优先级** | **P1** — AI chat stream + 任务进度推送 + 实时通知均为核心需求 |

### 2.6 文件存储抽象层 (P3)

| 项目 | 说明 |
|------|------|
| **现状** | MinIO + RustFS 作为数据库层存在 (`database/minio/`, `database/rustfs/`)，但无统一文件存储服务 |
| **影响** | 各模块直接操作 S3 客户端，无统一的上传/下载/预览/缩略图接口 |
| **建议** | 创建 `internal/services/storage/`，提供 `Storage` 接口 (Upload/Download/Delete/List/SignedURL)，底层可对接 MinIO/S3/Local |
| **优先级** | P3 — 需要文件上传场景时再实现 |

## 三、公共服务层缺失分析

### 3.1 文件系统服务 / Obsidian Vault 同步 (P1)

| 项目 | 说明 |
|------|------|
| **现状** | 无文件系统服务 |
| **影响** | Obsidian Vault 同步无法实现 |
| **建议** | `development-plan.md` Phase 4 已规划：`internal/services/filesystem/` — fsnotify 监听 + Markdown 解析 + 双向同步 |
| **优先级** | **P1** — Obsidian 场景的基石 |

### 3.2 搜索服务 (P1)

| 项目 | 说明 |
|------|------|
| **现状** | 无全文搜索能力，仅数据库 LIKE 查询 |
| **影响** | Obsidian 笔记全文搜索、知识检索均无法实现 |
| **建议** | `development-plan.md` Phase 4 已规划：`internal/services/search/` — 全文索引 + 标签/目录范围搜索。可选择 Bleve (纯 Go) 或 Elasticsearch |
| **优先级** | **P1** — Obsidian 知识搜索的核心依赖 |

### 3.3 AI 推理服务 (P1)

| 项目 | 说明 |
|------|------|
| **现状** | 完全未实现，但已在 `docs/module-design.md` 中完整设计 |
| **影响** | 所有 AI 能力 (对话/嵌入/流式输出) 均不可用 |
| **建议** | 实现 `internal/services/ai/`：AIProvider 接口 + Ollama/OpenAI 后端 + Chat/Embed/Stream API |
| **优先级** | **P1** — RAG 的核心推理引擎 |

### 3.4 通知服务 (P3)

| 项目 | 说明 |
|------|------|
| **现状** | 无邮件/短信/推送通知能力 |
| **影响** | 无法发送验证码、告警通知、任务完成通知 |
| **建议** | 创建 `internal/services/notification/`，提供 Email/SMS/Push 统一接口 |
| **优先级** | P3 — 按需实现 |

### 3.5 审计追踪服务 (P2)

| 项目 | 说明 |
|------|------|
| **现状** | 仅 `logger.InitAuditLog()` 写入文本文件，无结构化审计查询 |
| **影响** | 无法按用户/操作/时间范围查询审计记录 |
| **建议** | 增强现有审计日志为结构化服务，支持数据库存储 + API 查询 |
| **优先级** | P2 — 安全合规需求 |

### 3.6 模板引擎 (P3)

| 项目 | 说明 |
|------|------|
| **现状** | 无模板渲染能力 |
| **影响** | 邮件/报告/通知内容需硬编码 |
| **建议** | 集成 Go `text/template` + `html/template`，提供模板管理服务 |
| **优先级** | P3 — 通知服务的前置依赖 |

## 四、Obsidian RAG 可行性分析

### 4.1 场景定义

```
用户需求: 将 Obsidian Vault 中的 Markdown 笔记作为知识库，
通过自然语言问答检索相关内容，实现 RAG (Retrieval-Augmented Generation)。
```

### 4.2 所需组件链

```
┌─────────────────────────────────────────────────────────────┐
│                     Obsidian Vault                          │
│  (本地 Markdown 文件，front-matter + 双向链接 [[wiki]])      │
└─────────────────────────┬───────────────────────────────────┘
                          │ fsnotify 监听
                          ▼
┌─────────────────────────────────────────────────────────────┐
│ ① filesystem 服务                                          │
│  - fsnotify 文件变更监听                                     │
│  - Markdown 解析 (front-matter + body + [[链接]])            │
│  - 双向同步 (本地 ↔ 服务端)                                  │
└─────────────────────────┬───────────────────────────────────┘
                          │ 解析后的文档
                          ▼
┌─────────────────────────────────────────────────────────────┐
│ ② search 服务                                              │
│  - 全文索引 (bleve/elasticsearch)                            │
│  - 标签/目录范围过滤                                         │
│  - 结果排序 + 高亮                                          │
└─────────────────────────┬───────────────────────────────────┘
                          │ 检索结果
                          ▼
┌─────────────────────────────────────────────────────────────┐
│ ③ AI 服务 (Embedding + RAG)                                │
│  - 文档分块 (chunking)                                      │
│  - 向量嵌入 (Ollama/OpenAI Embedding)                       │
│  - 向量存储 (内存/向量数据库)                                │
│  - 检索 + 重排序                                            │
│  - LLM 生成 (Ollama/OpenAI Chat)                            │
└─────────────────────────┬───────────────────────────────────┘
                          │ 问答结果
                          ▼
┌─────────────────────────────────────────────────────────────┐
│ ④ s2 模块 — 知识管理模块                                    │
│  - Obsidian Vault 管理 API                                  │
│  - 笔记全文搜索 API                                         │
│  - RAG 问答 API (SSE 流式)                                  │
│  - 知识图谱 (标签关联 + 双向链接)                             │
│  - 知识库统计                                                │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 技术选型建议

| 组件 | 可选方案 | 推荐 | 理由 |
|------|----------|------|------|
| **Markdown 解析** | `gomarkdown`, `goldmark`, `blackfriday` | `github.com/yuin/goldmark` | 标准扩展支持 (GFM + front-matter + [[wiki]]) |
| **全文搜索** | Bleve (纯 Go), Elasticsearch, Meilisearch | Bleve | 纯 Go 无外部依赖，适合单体阶段 |
| **向量存储** | 内存 + 文件, Chroma, Milvus, Qdrant | 内存 + 文件 (起步) / Chroma (生产) | Obsidian 个人知识库规模小，内存足够 |
| **Embedding** | Ollama (nomic-embed-text), OpenAI (text-embedding-3) | Ollama 本地模型 | 兼顾隐私与成本 |
| **LLM** | Ollama (qwen2), OpenAI (GPT-4o) | Ollama 本地 + OpenAI fallback | `docs/module-design.md` 已规划路由策略 |
| **分块策略** | 固定大小, 语义分块, Markdown 结构分块 | Markdown 结构分块 | 按标题/段落分割，保持语义完整性 |
| **RAG 框架** | LangChain, LlamaIndex, 自建 | 自建 (轻量) | 依赖少，灵活可控 |

### 4.4 开发阶段建议

```
Phase A (当前 Phase 1 延展 — 2 周)
├── internal/services/filesystem/    — fsnotify + Markdown 解析
├── internal/services/search/        — Bleve 全文搜索
└── internal/services/ai/            — AIProvider + Ollama/OpenAI 后端 + Embedding

Phase B (Phase 2 — 1 周)
├── internal/services/ai/embedding   — 向量嵌入 + 内存向量存储
├── internal/services/ai/rag         — RAG 检索管道 (分块→嵌入→检索→重排序→生成)
└── internal/services/ws/            — SSE 流式输出

Phase C (Phase 3 — 1 周)
├── internal/modules/s2   — 增强为知识管理模块
├── Obsidian Vault 管理 API
├── 笔记全文搜索 API
├── RAG 问答 API (SSE)
└── 知识图谱 (标签关联 + 双向链接)
```

## 六、最终建议

### 6.1 优先实现路径

```
当前 → 立即开始 (4 周)
┌─────────────────────────────────────────────────────┐
│ 第 1-2 周:  AI 推理服务 (Ollama + OpenAI + Embedding) │
│            文件系统服务 (fsnotify + Markdown 解析)     │
│            全文搜索 (Bleve)                          │
├─────────────────────────────────────────────────────┤
│ 第 3 周:   WebSocket/SSE 实时推送                     │
│            AI Embedding + 向量存储 + RAG 管道          │
├─────────────────────────────────────────────────────┤
│ 第 4 周:   s2 增强为知识管理模块 (知识库 CRUD + 搜索)  │
│            RAG 问答 API + SSE 流式输出                 │
│            知识图谱 (标签关联 + 双向链接)               │
└─────────────────────────────────────────────────────┘
```

### 6.2 避免过度设计

- **搜索** 当前用 Bleve 纯 Go 实现，无需引入 ES 集群
- **向量存储** 个人知识库场景内存足够，先不上 Chroma/Milvus
- **RAG** 自建轻量管道，不上 LangChain 等重框架
- **AI** 先接 Ollama 本地模型，OpenAI 作为可选 fallback
