### 1. `internal/database` — 统一数据访问层（高优先级）

**设计参考**: `docs/module-design.md` §14

```go
// DBManager 全局管理器
type DBManager struct {
    Postgres map[string]*pgxpool.Pool
    MySQL    map[string]*sql.DB
    Redis    map[string]*redis.Client
    S3       map[string]*minio.Client
}

var DB *DBManager

func InitAll(cfg *config.Config) error  // 根据配置初始化所有连接
func CloseAll()                          // 优雅关闭
func Health() map[string]HealthStatus    // 聚合健康状态
```

**子包实现**:
| 子包 | 库 | 优先级 |
|------|----|--------|
| `database/mysql/` | `go-sql-driver/mysql` | **高**（主数据库） |
| `database/postgres/` | `pgx` / GORM | **高**（NAS 场景） |
| `database/redis/` | `go-redis/v9` | **高**（缓存/队列） |
| `database/minio/` | `minio-go/v7` | 中（对象存储） |
| `database/rustfs/` | `minio-go/v7` (S3 兼容) | 中（自建对象存储） |

**关键功能**:
- 连接池配置（max_open / max_idle / conn_max_lifetime）
- 多实例支持（main / nas 等多配置）
- 健康检查（ping + 超时）
- Prometheus 指标（open connections / query duration / health status）
- 在 `s0/health` 中聚合展示

**注意**: 需要先在 `config.yaml` 中完善 database 配置段（参考 `docs/module-design.md` §14 配置示例），并在 `internal/config/config.go` 中添加对应的 Config 结构体字段。

**需要在 go.mod 中添加依赖**: `github.com/jackc/pgx/v5`, `github.com/redis/go-redis/v9`, `github.com/minio/minio-go/v7`

## 第 3 周任务

> **核心**: 基础 services + 事件总线 + 调度器

### 5. `internal/services/eventbus` — 事件总线

**方案**: Redis Stream（模块间解耦的核心基础设施）

| 功能 | 说明 |
|------|------|
| 发布/订阅 | `Publish(topic, payload)` / `Subscribe(topic, handler)` |
| Redis Stream | 消息持久化 + 消费者组（多副本消费） |
| 本地同步 | 同进程内直接调用（go 函数调用，避免序列化开销） |
| 事件定义 | 统一 `Event` 结构体（ID / Topic / Payload / Timestamp / Source） |
| 典型事件 | `config.changed`, `user.created`, `ups.on_battery`, `scheduler.task_complete` |

**注意**: 设计时需考虑后续模块拆分后的事件序列化兼容性（protobuf 预留）。

**需要在 go.mod 中添加依赖**: `github.com/redis/go-redis/v9`（如 database 中已加则复用）

### 6. `internal/services/monitor` — 监控模块

| 功能 | 说明 |
|------|------|
| 健康检查聚合 | 集成 database / redis / 外部依赖的 health check |
| `/metrics` | Prometheus 指标暴露（HTTP 端点） |
| 系统指标 | Go runtime（goroutines / GC / memory） |
| 自定义指标 | 各模块通过 `metrics.Register()` 注册 |

**注意**: 指标暴露端点 `/metrics` 可通过 s0 模块路由，或在 monitor 模块独立注册。

**需要在 go.mod 中添加依赖**: `github.com/prometheus/client_golang`

### 7. `internal/services/cache` — 缓存管理

| 功能 | 说明 |
|------|------|
| 多级缓存 | 本地内存（freecache） + Redis 分布式缓存 |
| 防雪崩 | 随机过期时间 + 本地缓存兜底 |
| 防穿透 | 布隆过滤器（bloom filter） |
| 防击穿 | singleflight 请求合并 |
| 统一接口 | `Get(key)` / `Set(key, val, ttl)` / `Delete(key)` / `Exists(key)` |

**需要在 go.mod 中添加依赖**: `github.com/coocood/freecache`, `github.com/cespare/xxhash/v2`

### 8. `internal/services/scheduler` + `internal/services/jobs` — 任务调度

**设计参考**: `docs/module-design.md` §8

| 模块 | 功能 |
|------|------|
| `scheduler` | 定时/周期/一次性任务管理（robfig/cron + asynq） |
| `jobs` | 后台任务执行器（进度上报/取消/重跑/日志） |

**scheduler 核心模型**:
- Task（ID / Name / Type(cron\|once\|interval) / Expression / Handler / Payload / Status）
- TaskHandler 接口（Name / Execute）
- 全局处理器注册器 `RegisterHandler()`
- 任务持久化到数据库 + 启动时重新加载

**jobs 核心功能**:
- 任务进度上报（百分比 + 状态文字）
- 取消正在执行的任务（context cancellation）
- 失败自动重试（asynq 内置）
- 执行日志（TaskLog 表）

**需要在 go.mod 中添加依赖**: `github.com/robfig/cron/v3`, `github.com/hibiken/asynq`

## 测试规划

### 总体测试策略

参照 `docs/test.md` 定义的三层测试金字塔，Phase 1 所有模块测试遵循以下原则：

| 层级 | 比例 | 运行速度 | 依赖隔离 | 覆盖目标 |
|------|------|----------|----------|----------|
| **单元测试** | 70% | ms 级 | Mock 外部依赖（gomock / sqlmock / miniredis） | ≥ 70% 行覆盖 |
| **集成测试** | 20% | s 级 | dockertest / testcontainers（真实 MySQL/Redis） | 核心链路覆盖 |
| **组件测试** | 10% | 10s+ | 完整服务 + 依赖 | 健康检查 / 关键 API |

**构建标签隔离**:
```go
// 文件: xxx_integration_test.go
//go:build integration

// 文件: xxx_test.go  （默认，只跑单元测试）
```

```bash
# Makefile 命令（统一使用 scripts/test.sh）
make test-unit            # go test ./... -short -count=1 -race
make test-integration     # go test ./... -run Integration -count=1 -v -timeout 120s
make test-all             # go test ./... -count=1 -race -timeout 180s
make test-coverage        # go test ./... -count=1 -coverprofile=coverage.out
make test-pkg PKG=./internal/services/auth/...  # 指定包
```

**覆盖率门禁**: CI 中 `make test-unit` 失败则阻止合并。

```go
// internal/database/testutil/dockertest.go
package testutil

import (
    "github.com/ory/dockertest/v3"
    "github.com/ory/dockertest/v3/docker"
)

func RunMySQL(t *testing.T) (dsn string, close func()) {
    pool, err := dockertest.NewPool("")
    require.NoError(t, err)

    resource, err := pool.RunWithOptions(&dockertest.RunOptions{
        Repository: "mysql",
        Tag:        "8.0",
        Env:        []string{"MYSQL_ROOT_PASSWORD=test", "MYSQL_DATABASE=test"},
    }, func(config *docker.HostConfig) {
        config.AutoRemove = true
    })
    require.NoError(t, err)

    dsn = fmt.Sprintf("root:test@tcp(localhost:%s)/test?charset=utf8mb4&parseTime=True",
        resource.GetPort("3306/tcp"))

    pool.Retry(func() error {
        db, err := sql.Open("mysql", dsn)
        if err != nil { return err }
        return db.Ping()
    })

    close = func() { pool.Purge(resource) }
    t.Cleanup(close)
    return
}
```

### 测试驱动开发（TDD）建议

对于核心模块（auth / user / crypto / database），建议采用 TDD 流程：

```
1. 写接口定义（Service / Repository interface）
2. 写测试（gomock mock 依赖）
3. 运行测试 → 失败（RED）
4. 实现最小代码让测试通过（GREEN）
5. 重构优化（REFACTOR）
6. 补充集成测试（真实依赖）
```

这样可以确保：
- 接口设计先于实现，职责清晰
- 测试覆盖所有预期行为
- 重构时有安全网

## 风险与注意事项

1. **依赖冲突**: `go-sql-driver/mysql` 和 `pgx` 都使用 `database/sql` 接口，注意包路径和驱动注册
2. **asynq 需要 Redis**: scheduler 和 jobs 模块依赖 Redis，本地开发需确保 Redis 可用
3. **模块注册顺序**: `database.InitAll()` 必须在 `modules.InitAll()` 之前执行，确保 services 初始化时数据库已就绪
4. **环境差异**: 开发环境用 `GORM AutoMigrate` 快速迭代，生产用 `golang-migrate` 版本化迁移
5. **热加载一致性**: 配置变更时需确保所有相关模块原子性重载（eventbus 广播变更事件）
6. **Docker Compose 增强**: 当前 `docker-compose.yml` 只有 Go 后端，第 4 周需加入 MySQL + Redis 服务
7. **`cmd/migrate/` 和 `cmd/backup/` 入口**: 注意 `cmd/server/main.go` 与 CLI 入口的区别，避免互相引用导致 import cycle
